# Terminal Lab Usability QA

Purpose:
Review every beginner terminal lab from the perspective of someone with very little IT experience.

Goal:
The app should feel understandable, guided, and confidence-building.

---

# Core Beginner Questions

For EVERY scenario:

1. Does the user understand the problem?
2. Does the user know what to do next?
3. Does the user understand WHY the command matters?
4. Does the output make sense?
5. Does the user understand whether they succeeded?
6. Does the next step feel obvious?
7. Could a non-technical person follow this?

---

# Beginner Flow Standard

Every scenario should ideally contain:

## Problem
What happened?

## Meaning
What does this usually indicate?

## First Step
What should we check first?

## Command
Which command helps investigate this?

## Explanation
What does the output mean?

## Resolution
What was the issue?

## Takeaway
What skill did the learner just use?

---

# Desktop UX Checklist

## Layout
- Main task visible without scrolling
- Terminal visible immediately
- Controls grouped logically
- Reading order obvious
- No visual overload

## Control Clarity
- Commands = examples
- Hint = next step
- Guide = walkthrough
- Ask Coach = explanation help

## Input UX
- Input field obvious
- Placeholder helpful
- Run button visible
- Output readable

## Cognitive Load
- No unnecessary panels visible
- No advanced jargon early on
- One clear objective at a time

---

# Mobile UX Checklist

## Controls
- Large tap targets
- Keyboard not obstructive
- Important controls reachable with thumbs

## Layout
- No side-scrolling
- Terminal readable
- Current task always visible

## Beginner Guidance
- User never feels trapped
- Help always accessible
- Walkthrough discoverable

---

# Scenario Quality Checklist

## Clarity
- Short sentences
- Real-world wording
- Minimal jargon

## Learning Design
- One new concept at a time
- Reinforcement repetition
- Increasing difficulty gradually

## Confidence Building
- Positive reinforcement
- Clear success states
- Helpful failure messages

# Architecture Rule

Do NOT replace the terminal engine.

Do:
- progressively simplify
- layer improvements
- hide complexity
- improve onboarding
- improve wording
- improve flow

Avoid:
- rebuilding the app
- removing core systems
- over-gamifying
- replacing the scenario engine
